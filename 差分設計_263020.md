1. サンプルシート入力対応

いちばん価値が高いです。

今は FASTQ 名から自動認識ですが、実運用では

サンプル名を手で揃えたい

群情報を先に持ちたい

layout を明示したい

将来 reporter 側に渡したい

が出てきます。

入れる項目の最小案はこれです。

sample_id

r1_path

r2_path

layout

group

note（任意）

まずは
「自動認識」か「CSV読込」かを選べる
ようにするだけで十分強いです。

2. group / condition 列の導入

これは次の reporter や DEG 側につながります。

たとえば sample table に

control

treated
のような列を持てるだけで、後工程が一気に楽になります。

今は解析自体には使わなくてよくて、
保存だけする ので十分です。

3. 出力形式の拡張

次に効くのはここです。

今ある:

transcript_quant.csv

gene_quant.csv

次にあると嬉しい:

gene_numreads.csv

transcript_numreads.csv

sample_qc_summary.csv

特に sample_qc_summary.csv は、今画面で出している QC 表をそのまま保存できるので価値が高いです。

4. 失敗サンプルを含む部分成功対応

複数サンプルになったので、ここが次の壁です。

欲しい動き:

2サンプル中1サンプル失敗でも run 自体は継続

成功サンプルだけで集約

失敗サンプルは結果画面で明示

run.log に理由保存

これは実運用でかなり重要です。

いちばんおすすめ

次の1本を選ぶなら、

v0.1.5: サンプルシート入力対応

がベストです。

group 情報の入口になる

UI の見た目をいじらず機能拡張できる

からです。
