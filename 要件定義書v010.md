# iwa-rnaseq-counter 要件定義たたき台

## 1. 背景

研究所において、RNA-Seq の初期解析（FASTQ から count / quant 出力まで）は依然として dry 側への依頼が前提になりやすい。結果として、wet 側の研究者が自分で素早く解析に着手できず、試行錯誤の速度が落ちる。

本アプリ **iwa-rnaseq-counter** は、まず **wet の人が自力で FASTQ から count まで到達できること** を最優先目標とする。将来的には関連するサブアプリ群を追加し、RNA-Seq の周辺解析を段階的に支援する。

## 2. 目的

本アプリの目的は以下とする。

1. wet 研究者が GUI 操作を通じて、FASTQ から gene-level count / transcript quant を出力できること
2. dry 研究者に依頼しなくても、最低限の前処理・定量結果を自力で得られること
3. 実行条件・参照・ログを保存し、再現性を確保すること
4. 将来的なサブアプリ連携のため、拡張しやすい構造を持つこと

## 3. 想定ユーザー

### 主対象

* Linux やコマンドラインに不慣れな wet 研究者
* RNA-Seq 解析を「まず一回まわしたい」利用者
* FASTQ から gene count / transcript quant を自力で取得したい利用者

### 副対象

* wet 研究者を支援する dry 研究者
* 参照設定や実行条件をテンプレート化したい管理者

## 4. スコープ

### 本バージョンで対応する範囲

* FASTQ ファイルの検出
* single-end / paired-end 判定
* lane 検出と論理統合
* 参照ファイル選択または index 新規作成
* 以下の解析モード実行

  * STAR + featureCounts
  * Salmon
  * kallisto
* transcript quant 出力
* gene-level 集約出力
* strandedness の自動推定および手動上書き
* CPU / メモリに基づく並列実行制御
* ログ、設定、出力ファイルの整理保存

### 本バージョンで対応しない範囲

* Differential Expression 解析
* QC レポートの高度な可視化
* MultiQC など統合 QC ダッシュボード
* GO / Pathway 解析
* クラウド実行
* Docker / Kubernetes 前提の実行
* LIMS 連携

## 5. 製品コンセプト

**「wet の人が、まず count まで自力で出せる RNA-Seq カウンター」**

本アプリは、AI がすべてを自律判断することを目的としない。むしろ、GUI と自動判定を組み合わせて、事故を減らしながら wet ユーザーを解析完了まで導く **実用的な実行支援アプリ** とする。

## 8. 機能要件

### 8.1 入力 FASTQ 検出

* ユーザーがホスト内ディレクトリを指定できる
* 指定ディレクトリ以下の FASTQ / FASTQ.GZ を検出できる
* サンプル名候補を抽出できる
* single-end / paired-end を自動判定できる
* 判定結果を一覧表示できる
* ユーザーが手動修正できる

### 8.2 lane 検出・統合

* L001 / L002 などの lane 分割ファイルを検出できる
* lane は物理結合せず、論理的に同一サンプルとして束ねる
* lane 検出結果を画面に表示する

### 8.3 解析モード

以下の 3 系統をサポートする。

1. alignment count

   * STAR
   * featureCounts
2. transcript quant

   * Salmon
3. transcript quant

   * kallisto

### 8.4 gene-level 集約

* Salmon / kallisto の transcript-level 出力から gene-level 行列を生成する
* transcript-level 出力も保持する
* gene-level 出力を wet ユーザーが扱いやすい形式で保存する

### 8.5 strandedness

* デフォルト値は Auto-detect とする
* 手動上書きを可能とする
* 推定結果、信頼度、根拠を表示する
* 推定不能時は警告し、必要に応じて手動指定を促す

### 8.6 参照・index 管理

* 既存 index があれば選択利用できる
* index がなければ新規作成できる
* 保存先は以下から選択可能とする

  * 共通参照棚（デフォルト）
  * プロジェクト内
* 参照セットの再利用を可能にする

### 8.7 リソース制御

* システムの CPU 数とメモリ量を表示する
* 解析方式ごとの推定 CPU / メモリ使用量を表示する
* 並列数をスライダーで指定できる
* 上限超過時は警告を表示し、RUN を不可にする
* デフォルトは 1 ジョブ逐次実行とする

### 8.8 実行・進捗表示

* RUN ボタンにより解析を開始できる
* 現在実行中のステップを表示できる
* 標準出力 / 標準エラーの要約を表示できる
* 実行完了後に出力ファイル一覧を提示できる

### 8.9 ログ・再現性

* 実行設定を JSON 等で保存する
* 実行コマンドを保存する
* 使用参照情報を保存する
* ソフトウェアバージョンを保存する
* stdout / stderr を保存する
* 出力先ディレクトリを整理して保存する

## 9. 非機能要件

### 9.1 使いやすさ

* コマンドライン操作を前提としない
* wet ユーザーが一画面ずつ順に進める UI とする
* 誤設定時には、専門用語だけでなく平易な警告を表示する

### 9.2 再現性

* 同一入力・同一参照・同一設定で同一結果が再現しやすい構造とする
* Pixi lock により環境再現性を担保する

### 9.3 保守性

* サブアプリ追加を見据えてモジュール分割する
* UI、実行、参照管理、推定ロジックを分離する

### 9.4 可搬性

* Pixi が導入可能な環境で持ち運べること
* 特定クラスタやコンテナ基盤に依存しないこと

## 11. RUN 可否条件

### 11.1 全体方針

RUN は以下をすべて満たした場合のみ可能とする。

* FASTQ が認識されている
* サンプル構造が整合している
* 必要参照が揃っている
* index が存在する、または対象バージョンで新規作成可能である
* strandedness が推定可能、または手動指定されている
* 並列設定が CPU / メモリ上限内である、または対象バージョンで簡略警告条件を満たしている
* 出力先に書き込み可能である

### 11.2 v0.1.0 での RUN 可否条件

v0.1.0 では、以下をすべて満たした場合のみ RUN 可能とする。

* FASTQ が認識されている
* サンプル構造が整合している
* 既存 Salmon index が指定され、有効である
* tx2gene ファイルが指定され、有効である
* strandedness が自動推定済み、または手動指定されている
* 出力先に書き込み可能である

v0.1.0 では、新規 index 作成は RUN 条件に含めない。
また、CPU / メモリについては詳細な並列制御ではなく、情報表示と簡略警告を行う。

## 16. MVP 定義

### 16.1 MVP の目的

MVP（Minimum Viable Product）の目的は、**wet ユーザーが FASTQ から gene-level count または transcript quant を GUI 操作だけで出力できることを、事故を抑えながら実現すること** とする。

MVP は「全部入り」ではなく、以下を最優先とする。

1. 初回利用者でも迷いにくいこと
2. RNA-Seq の基本的な入口として成立すること
3. 致命的な事故（pair 判定ミス、strandedness ミス、過剰並列）を減らすこと
4. dry 側が後から追跡できるログと設定を残すこと

### 16.2 MVP に含める機能

#### A. 入力とFASTQ認識

* 入力ディレクトリ指定
* 出力ディレクトリ指定
* FASTQ / FASTQ.GZ の検出
* single-end / paired-end 自動判定
* lane 検出と論理統合
* サンプル一覧表示
* ユーザーによる手動修正

#### B. 解析実行

* STAR + featureCounts 実行
* Salmon 実行
* kallisto 実行
* transcript quant から gene-level 集約出力

#### C. 参照管理

* 既存 index の利用
* index 未作成時の新規作成
* 共通参照棚をデフォルト利用
* オプションでプロジェクト内保存

#### D. 安全機構

* strandedness: Auto-detect をデフォルト
* strandedness の手動上書き
* 推定結果の画面表示
* CPU / メモリ取得
* 並列数スライダー
* 上限超過時の RUN 無効化

#### E. 実行結果管理

* 実行ログ保存
* 実行設定保存
* 参照情報保存
* ソフトウェア version 保存
* 出力ファイル一覧表示

### 16.3 MVP で含めない機能

以下は MVP では対応しない。

* Differential Expression 解析
* 統合 QC ダッシュボード
* GO / Pathway 解析
* 会話型エージェントUI
* 複数ユーザー同時利用を前提としたジョブスケジューラ
* 高度なジョブ再開制御
* クラウド・分散実行
* wet 向け自然言語レポート生成

### 16.4 MVP の操作フロー

MVP の基本操作フローは以下とする。

1. 入力ディレクトリを指定する
2. FASTQ をスキャンする
3. サンプル構造（single/paired、lane）を確認・必要なら修正する
4. 解析モードを選択する
5. 参照と index を選択または作成する
6. strandedness と並列数を確認する
7. RUN を実行する
8. count / quant / gene-level 出力を確認する

### 16.5 MVP の成功基準

MVP が成功したと見なす条件は以下とする。

1. wet ユーザーが、CLI を使わずに FASTQ から count または quant を出力できる
2. paired-end / lane / strandedness の基本的な事故を UI で減らせる
3. 並列設定の過剰指定を防げる
4. 実行設定とログを dry 側が追跡できる形で保存できる
5. 初回版として GitHub 公開可能なまとまりを持つ

### 16.6 MVP の優先順位

#### Must Have

* pair / lane 判定
* STAR + featureCounts

#### Should Have

#### Could Have

* Ollama を使った実行前チェック説明
* ログ要約
* wet 向け簡易説明文

### 16.7 MVP の初期実装順

MVP の初期実装は以下の順を推奨する。

1. FASTQ 検出とサンプル構造認識
2. STAR + featureCounts 実行
3. Salmon / kallisto 実行
4. gene-level 集約
5. 参照管理と index 作成

### 16.8 MVP の位置づけ

MVP は最終完成版ではなく、**wet の人が dry に依頼しなくても最初の一歩を進められる状態を作るための入口版** と位置づける。

## 17. v0.1.0 初回リリース定義

### 17.1 位置づけ

v0.1.0 は、iwa-rnaseq-counter の**最初の実用版**とする。

目的は、wet ユーザーが迷いすぎず、重すぎる構成に足を取られず、まず **FASTQ から transcript / gene quant を安全に出力できること** に絞る。

そのため v0.1.0 では、解析エンジンや参照管理を意図的に絞り込み、成功率と使いやすさを優先する。

### 17.2 v0.1.0 の基本方針

* 解析エンジンは **Salmon のみ** とする
* STAR + featureCounts, kallisto は将来拡張を見据えて UI 上にプレースホルダーを残すが、v0.1.0 では実行不可とする
* Salmon index は **既存の利用のみ** を対象とする
* strandedness 自動判定は、Salmon の自動ライブラリ判定機能を活用した中核機能として扱う
* wet ユーザーが最終的に得たい **transcript-level quant / gene-level quant の CSV 出力** を最優先とする

### 17.3 v0.1.0 に含める機能

#### A. 解析エンジン

* Salmon を用いた transcript-level quant
* Salmon の結果から gene-level 集約を実施
* transcript-level / gene-level の両方を CSV で出力
* CSV では、少なくとも `TPM` と `NumReads` 相当を保持できる構造を想定する

#### B. データ認識

* FASTQ / FASTQ.GZ 検出
* single-end / paired-end 自動判定
* lane 検出
* lane の論理統合
* サンプル一覧表示
* 必要最小限の手動修正

#### C. 参照管理

* 既存 Salmon index の選択
* transcript-to-gene 対応表の指定
* 必要に応じた annotation 関連ファイルの指定
* 新規 index 作成は非対応

#### D. 安全装置

* strandedness 自動判定
* strandedness 推定結果の表示（信頼度・根拠含む）
* 必要に応じた手動上書き
* サンプル構造不整合時の警告
* 実行前バリデーション
* CPU / メモリ情報の表示と簡略警告

#### E. 出力

* transcript-level quant CSV
* gene-level quant CSV
* 実行設定ファイル
* 実行ログ
* 最低限の run summary

### 17.4 v0.1.0 で含めない機能

以下は v0.1.0 では対応しない。

* STAR + featureCounts
* kallisto
* 新規 index 作成
* 複数解析エンジン切り替え
* 高度なリソース見積もりと並列ジョブ制御
* DEG 解析
* QC レポート統合表示
* Ollama を用いた会話型ナビゲーション

### 17.5 v0.1.0 の価値提案

v0.1.0 の価値は、機能の多さではなく、**wet ユーザーが最初にぶつかる壁を最小構成で越えられること** にある。

特に以下を価値の中心とする。

1. FASTQ から quant まで進める

3. 重すぎる構成を避け、導入ハードルを下げる
4. 後続バージョンで STAR 系へ拡張しやすい土台を持つ

### 17.6 v0.1.0 成功基準

v0.1.0 が成功とみなせる条件は以下とする。

1. wet ユーザーが GUI から Salmon 実行を完了できる

4. transcript / gene の quant CSV を出力できる

### 17.7 v0.1.0 の実装優先順位

## 18. v0.1.0 画面設計

### 18.1 画面設計の基本方針

v0.1.0 の画面は、**wet ユーザーが上から順に見ていけば実行まで到達できること** を最優先とする。

そのため、複雑な画面遷移よりも、1画面内で以下の流れが自然に進む構成を基本とする。

1. 入力場所を決める
2. FASTQ を認識させる
3. サンプル構造を確認する
4. 参照を選ぶ
5. strandedness を確認する
6. 実行する
7. 出力を確認する

### 18.2 画面構成案

v0.1.0 では、**シングルページ構成 + セクション分割** を基本案とする。

画面は以下の 6 セクションで構成する。

1. 解析情報
2. 入力 FASTQ
3. サンプル認識結果
4. 参照設定
5. 実行設定
6. 実行・結果

### 18.3 セクション詳細

#### セクション1: 解析情報

目的: run 全体の識別情報を入力する

表示項目:

* 解析名
* 入力ディレクトリ
* 出力ディレクトリ
* 「FASTQ をスキャン」ボタン

補助表示:

* 入力ディレクトリの存在確認
* 出力ディレクトリの書き込み可否
* スキャン対象ファイル数

メッセージ例:

* 「入力ディレクトリを指定してください」
* 「出力先に書き込み可能です」
* 「FASTQ を 12 件検出しました」

#### セクション2: 入力 FASTQ

目的: 認識対象ファイルの一覧を利用者に見せる

表示項目:

* FASTQ 一覧テーブル

  * file name
  * path
  * size
  * detected suffix

補助機能:

* 再スキャン
* 検出 0 件時の警告表示

メッセージ例:

* 「FASTQ / FASTQ.GZ を検出しました」
* 「対象ファイルが見つかりませんでした」

### 18.5 UI の文言方針

wet ユーザー向けに、文言は以下を基本とする。

* 専門用語だけで終わらせない
* 英語ファイル名はそのまま表示しつつ、日本語で意味を補う
* エラー文は「何が不足しているか」「どう直せばよいか」を短く示す

例:

* 悪い例: `invalid library type`
* 良い例: `ライブラリの向き（strandedness）を判定できませんでした。必要なら手動で指定してください。`

* DEG / QC / GO の結果表示UI

補足として、将来拡張を示すために STAR + featureCounts および kallisto のプレースホルダー表示は許容する。ただし v0.1.0 では選択不可または実行不可とする。

### 18.9 RUN ボタンの表示方針

* 実行可能時: 有効化
* 実行不可時: 無効化し、直上に理由を表示する
* 実行中: 二重実行を防ぐため無効化する

RUN 不可理由の表示例:

* `FASTQ が検出されていません`
* `Salmon index が指定されていません`
* `tx2gene ファイルが指定されていません`
* `サンプル構造に修正が必要です`

### 18.10 初回実装での見た目方針

* シンプル
* 情報を詰め込みすぎない
* 表示色は status の区別に限定する
* wet ユーザーが「次に何をすればよいか」をすぐ理解できることを優先する

## 20. v0.1.0 データ構造定義

### 20.1 目的

本章では、v0.1.0 の実装で中心となるデータ構造を定義する。

目的は以下の通り。

* UI と内部ロジックの受け渡しを明確にする

* 関数間の入出力を安定させる

### 20.2 `fastq_df`

FASTQ 検出結果を UI 表示用に保持する DataFrame とする。

想定列:

* `file_name`: ファイル名
* `path`: フルパス
* `size_bytes`: ファイルサイズ
* `extension`: 拡張子
* `is_gzipped`: gzip 圧縮有無
* `detected_sample_token`: サンプル名候補
* `detected_read_token`: R1 / R2 / None
* `detected_lane_token`: L001 など / None

用途:

* 入力 FASTQ 一覧表示
* サンプル構造推定の前処理

### 20.3 `sample_groups`

FASTQ 群を sample 単位に束ねた辞書とする。

想定構造:

```python
{
    "SampleA": {
        "files": [
            "/data/SampleA_L001_R1.fastq.gz",
            "/data/SampleA_L001_R2.fastq.gz",
            "/data/SampleA_L002_R1.fastq.gz",
            "/data/SampleA_L002_R2.fastq.gz"
        ],
        "read_pairs": {
            "L001": {"R1": "...", "R2": "..."},
            "L002": {"R1": "...", "R2": "..."}
        }
    }
}
```

用途:

* SE / PE 判定
* lane 論理統合
* Salmon 実行用入力整理

各 sample の single-end / paired-end 推定結果を保持する辞書とする。

## 21. v0.1.0 内部モジュール設計: 関数一覧・引数・返り値

### 21.1 設計方針

v0.1.0 では、関数名はできるだけ以下の規則で統一する。

* 取得: `get_...`
* 検出: `discover_...` / `detect_...`
* 推定: `infer_...`
* 検証: `validate_...`
* 実行: `run_...`
* 構築: `build_...`
* 集約: `aggregate_...`
* 描画: `render_...`
* 初期化: `init_...`

また、返り値は v0.1.0 ではできるだけ **辞書または DataFrame に寄せる** ことで、UI との接続を単純化する。

### 21.11 返り値フォーマットの共通方針

バリデーション系関数は、できるだけ以下の共通形式に揃える。

```python
{
    "is_valid": bool,
    "errors": list[str],
    "warnings": list[str]
}
```

実行系関数は、できるだけ以下の情報を含める。

```python
{
    "is_success": bool,
    "errors": list[str],
    "warnings": list[str],
    "log_summary": str,
    "outputs": dict | list
}
```
